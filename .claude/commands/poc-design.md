# PoC Design

## 目的
`.codex/docs/USERS_GUIDE.md` に記載した利用者操作と期待体験について、正式な要件定義やアーキテクチャ設計に入る前に、最小実装で使い勝手を検証するための `.codex/docs/POC.md` を設計する。

## 先に読むもの
- `CLAUDE.md`
- `.codex/docs/MASTER.md`
- `.codex/docs/RESEARCH.md`
- `.codex/docs/USERS_GUIDE.md`
- 既存の `.codex/docs/POC.md` がある場合はその内容
- `.codex/docs/REQUIREMENTS.md`、`.codex/docs/ADR.md`、`.codex/docs/ARCHITECTURE.md` が既に存在する場合は、それらも読む

PoC 設計は必ず `USERS_GUIDE.md` に記載された利用者操作と `RESEARCH.md` に整理された技術前提を根拠に行うこと。不足している体験や仕様を推測して補わないこと。

## このスキルで判断すること
1. `USERS_GUIDE.md` のうち、どの操作フローや画面体験を先に PoC で試すべきか。
2. その PoC で何を観察すれば「使い勝手の良し悪し」を判断できるか。
3. PoC の対象外に切り捨てるものは何か。
4. PoC の成功条件、実装方針、実行手順、観察方法をどう定義するか。
5. PoC の結果を `REQUIREMENTS.md` にどう還元するか。

## PoC 実施対象の判断基準
次のいずれかに該当する操作や体験は、PoC の優先対象とする。

- ユーザーが実際に触ったときの操作手順や反応速度が、文書だけでは妥当か判断しづらい。
- 表示、操作、フィードバックの組み合わせが多く、机上で決めると手戻りが大きい。
- 技術的には成立しそうでも、利用者にとって分かりにくい可能性がある。
- 早期に使い勝手を見ないと、後続の `REQUIREMENTS.md` が机上要件に寄りすぎる。

次の条件をすべて満たすものは、PoC の対象外にしてよい。

- 使い勝手に影響しない内部処理だけの論点である。
- `USERS_GUIDE.md` 上でも操作が単純で、誤解の余地が少ない。
- PoC を作らなくても、後続の `REQUIREMENTS.md` に直接落として問題ない。

## 手順
1. `.codex/docs/RESEARCH.md` から技術制約、使用予定のミドルウェア、外部依存、既知の制約を確認する。
2. `.codex/docs/USERS_GUIDE.md` から利用者操作、期待結果、迷いやすい箇所、フィードバック要求を抽出する。
3. 操作フローを分解し、「PoC で触るべき部分」と「今回は捨てる部分」を切り分ける。
4. 各 PoC 対象について、何を観察すれば使い勝手を評価できるかを定義する。
5. 各対象について、最小実装の範囲、ダミーでよい部分、実データが必要な部分を整理する。
6. 実行手順、観察方法、成功条件、失敗時の見直し方を定義する。
7. PoC の結果を `REQUIREMENTS.md` にどのように反映するかを整理する。
8. 結果を `.codex/docs/POC.md` に反映できる形でまとめる。

## 必須出力
PoC 設計の結果には、少なくとも次を含める。

- PoC の目的
- PoC で検証する利用者操作一覧
- 各操作を選んだ理由
- PoC の対象外にするもの
- PoC の成功条件
- PoC の実現手段
- PoC の実装方針
- PoC コードの実行コマンド
- 観察方法
- PoC 結果を `REQUIREMENTS.md` に反映する観点

## `docs/POC.md` に記載すべき観点

### 1. PoC の目的
- 何の使い勝手を検証する PoC なのかを明記する。
- `USERS_GUIDE.md` のどの操作フローに対応するかを書く。

### 2. 検証対象の利用者操作
- どの操作フローを実際に触って確認するかを列挙する。
- 各項目について「なぜそれを先に触る必要があるか」を書く。
- 対象外にする操作も明記する。

### 3. 成功条件
- 観測可能で判定可能な条件にする。
- 「ユーザーが迷わず目的操作を完了できる」「必要な状態変化が画面やログで確認できる」など、使い勝手に紐づける。
- 「動いたら成功」のような曖昧な書き方を避ける。

### 4. 実現手段・実装方針
- PoC の対象範囲を必要最小限に絞る。
- 本番品質ではなく、検証に必要な最小構成で使い勝手を確認する方針を書く。
- ダミー実装、簡略化したシーン、手動操作、記録済みデータの再生など、近道を明示する。
- どの入出力だけを本物として扱うかを書く。

### 5. 実行コード・観察方法
- 実行コマンドをそのまま再現できる形で書く。
- 何を見て使い勝手を判断するか、画面、ログ、操作回数、待ち時間、迷いポイントなどの観点で明記する。

### 6. REQUIREMENTS への反映方針
- PoC で得た学びを、後続の `REQUIREMENTS.md` にどの粒度で反映するかを書く。
- 削るべき機能、追加すべきフィードバック、制約として固定すべき事項を整理する。

## 注意事項
- PoC は `USERS_GUIDE.md` に記載された体験の検証を目的とし、本番実装の代替にしてはいけない。
- PoC コードは本番コードへ流用しないことを明記すること。
- PoC のために入れた近道、ダミー、手動操作を、そのまま本番要件へ持ち込まないこと。
- PoC の結果、`USERS_GUIDE.md` の操作や期待体験に問題が見つかった場合は、`REQUIREMENTS.md` へ進む前に `USERS_GUIDE.md` を更新すること。
- 実装詳細を書きすぎて `POC.md` を設計書の代替にしないこと。

## やってはいけないこと
- `USERS_GUIDE.md` にない操作を勝手に追加して PoC 範囲を広げること。
- 使い勝手検証が目的なのに、内部アーキテクチャの美しさを PoC の主目的にすり替えること。
- PoC の成功条件を曖昧な主観評価で定義すること。
- PoC のサンプル実装を、そのまま本番実装へ昇格させる前提で書くこと。

## 推奨フォーマット

```md
# POC.md

## PoC の目的
- 対象フロー:
- 検証したい使い勝手:

## 検証対象
### 1. <利用者操作>
- 確認したい使い勝手:
- この操作を先に検証する理由:
- 成功条件:
- 実現手段:
- 実装方針:
- 実行コマンド:
- 観察方法:

## 対象外
- 今回は検証しないもの:

## REQUIREMENTS への反映
- 固定する事項:
- 見直す事項:

## 注意事項
- PoC コードは本番コードに流用しない。
```
