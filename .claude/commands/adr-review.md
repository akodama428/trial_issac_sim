# ADR Review

## 目的
`.codex/docs/ADR.md` をレビューし、要件適合性と設計品質の不足を明確にする。

## 先に読むもの
- `CLAUDE.md`
- `.codex/docs/MASTER.md`
- `.codex/docs/PROJECT.md`
- `.codex/docs/ADR.md`

関連文書が不足している場合は、その不足自体を指摘事項として扱う。

## レビュー時の更新ルール
- 指摘事項が 0 件になった場合は、`.codex/docs/ADR.md` の frontmatter の `status` を `承認済` に更新する。
- 指摘事項が 1 件以上ある場合は、`.codex/docs/ADR.md` の `status` を勝手に承認状態へ変更してはいけない。

## レビュー観点

### 1. クリーンアーキテクチャ
最低限、次を確認する。
- 単一責務の原則
- 依存関係ルール

確認ポイント:
- 各主要コンポーネントに主たる変更理由が1つに絞られているか
- UI、アプリケーション、ドメイン、インフラの責務が混ざっていないか
- ドメインや中核ロジックがフレームワークやインフラ詳細に直接依存していないか
- 外側の詳細が内側のポリシーに依存する方向になっているか

### 2. 可読性
確認ポイント:
- 文書構成が分かりやすいか
- コンポーネント、境界、データフローが追いやすいか
- 用語が一貫しているか
- 抽象論だけで終わらず、実装イメージに落ちる具体性があるか

### 3. 設計根拠
確認ポイント:
- なぜその構成を選ぶのかが説明されているか
- 代替案との比較があるか
- メリット・デメリットが記載されているか
- トレードオフが明示されているか
- 要件定義番号とモジュールの対応表があるか

### 4. 要件適合性
確認ポイント:
- `.codex/docs/PROJECT.md` に書かれた要件を満たす設計になっているか
- MVP の主要機能が設計に反映されているか
- 前提条件やスコープ外の扱いが設計と矛盾していないか

## 出力ルール
- レビューは指摘事項を優先する。
- 重要度の高い順に並べる。
- 各指摘で、どの文書のどの記述に対する指摘かを明示する。
- 指摘ごとに「何が問題か」「なぜ問題か」「どう直すべきか」を簡潔に書く。
- 指摘がない場合でも、その旨を明記し、残留リスクや未確認事項があれば添える。
- 指摘事項が 0 件だった場合は、`.codex/docs/ADR.md` の frontmatter の `status` を `承認済` に更新したことを明記する。

## 推奨フォーマット

```md
## Findings
1. [重大度] 指摘タイトル
対象: <文書名 / セクション>
問題: <何が不足または不整合か>
理由: <なぜ問題か>
修正案: <どう直すべきか>

## Open Questions
- <必要なら記載>

## Summary
- <全体評価を短く記載>
```

## レビュアーとしての振る舞い
- 褒めることより、設計上の欠陥、曖昧さ、将来の破綻要因を優先して見つける。
- 要件未充足と責務混在は強く指摘する。
- 逆に、十分に妥当な点は短く触れるに留める。
