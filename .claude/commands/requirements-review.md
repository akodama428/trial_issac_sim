# Requirements Review

## 目的
要件定義文書をレビューし、ビジョン、課題、主要機能、要件定義のつながりと不足を明確にする。

## 先に読むもの
- `CLAUDE.md`
- `.codex/docs/MASTER.md`
- `.codex/docs/PROJECT.md`
- `.codex/docs/USERS_GUIDE.md`

関連文書が不足している場合は、その不足自体を指摘事項として扱う。

## レビュー時の更新ルール
- `.codex/docs/PROJECT.md` の `## 要件定義` に並ぶ要件には、要件番号を連番で付与する。
- 要件番号は上から順に一意に振る。
- 既存の要件番号が欠番、重複、表記ゆれしている場合は、レビュー時に連番へ整理する。
- 要件番号の形式は、文書内で一貫していればよいが、基本は `REQ-001`, `REQ-002` のようなゼロ埋め連番を推奨する。
- レビュー結果の指摘事項が 0 件になった場合は、`.codex/docs/PROJECT.md` の frontmatter の `status` を `承認済` に変更する。
- 指摘事項が 1 件以上ある場合は、`status` を勝手に `承認済` に変更してはいけない。

## レビュー観点

### 1. ビジョン・課題と主要機能の整合
確認ポイント:
- ビジョンに対して主要機能が妥当か
- 解決する課題に対して主要機能が妥当か
- 主要機能が課題解決に直接つながっているか
- 主要機能に抜け漏れや過剰実装がないか

### 2. 主要機能と要件定義の整合
確認ポイント:
- 主要機能に対して要件定義が十分に展開されているか
- 各要件がどの主要機能を支えるかが分かるか
- 要件が抽象的すぎず、実装や検証に落とせる粒度か
- 要件間に重複、矛盾、抜けがないか

### 3. 要件定義の階層構造
確認ポイント:
- ビジョン → 課題 → 主要機能 → 要件定義 の階層が自然につながっているか
- 上位概念と下位要件が混在していないか
- 要件の粒度が揃っているか
- 説明、方針、具体要件が混線していないか

### 4. 正常系・異常系
確認ポイント:
- 正常系の利用シナリオが要件に反映されているか
- 入力なし、該当なし、調査失敗、データ欠損などの異常系が考慮されているか
- エラー時や未該当時の期待動作が明記されているか

### 5. 要件定義レビューとして追加で見るべき観点
確認ポイント:
- 要件が測定可能または判定可能か
- 前提条件、制約、スコープ外が要件と矛盾していないか
- 用語が一貫しているか
- 非機能要件や運用条件の不足がないか
- 後続の設計やテストへ受け渡せる記述になっているか

### 6. USERS_GUIDE と要件の整合
確認ポイント:
- `.codex/docs/USERS_GUIDE.md` に記載すべき利用者操作が、`.codex/docs/PROJECT.md` の要件で支えられているか
- 利用者が実施するセットアップ、生成、参照、更新、基本確認の流れに必要な要件が不足していないか
- `USERS_GUIDE.md` にしか存在しない重要機能や前提条件がないか
- 要件の正常系・異常系が、利用手順上の期待動作と矛盾していないか

## 出力ルール
- レビューは指摘事項を優先する。
- 重要度の高い順に並べる。
- 各指摘で、どの記述に対する指摘かを明示する。
- 指摘ごとに「何が問題か」「なぜ問題か」「どう直すべきか」を簡潔に書く。
- 指摘がない場合でも、その旨を明記し、残留リスクや未確認事項があれば添える。
- 要件番号を付与または整理した場合は、そのことを要約に含める。
- 指摘事項が 0 件だった場合は、`PROJECT.md` の `status` を `承認済` に更新したことを明記する。

## 推奨フォーマット

```md
## Findings
1. [重大度] 指摘タイトル
対象: <文書名 / セクション>
問題: <何が不足または不整合か>
理由: <なぜ問題か>
修正案: <どう直すべきか>

## Open Questions
- <必要なら記載>

## Summary
- <全体評価を短く記載>
```

## レビュアーとしての振る舞い
- 要件不足、曖昧さ、矛盾、検証不能な記述を優先して見つける。
- 主要機能と要件の対応が弱い場合は強く指摘する。
- 好意的に補完せず、文書に書かれていない内容は不足として扱う。
- `USERS_GUIDE.md` に現れる利用者操作を実現できるだけの要件が `PROJECT.md` にあるかを必ず確認する。
