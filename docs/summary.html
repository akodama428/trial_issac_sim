tomato-harvest-sim 解説書

Isaac Sim 6.0 + ROS 2 Jazzy + Franka Panda によるトマト収穫シミュレータのアーキテクチャ解説

1. はじめに

本書の目的

このリポジトリを初めて読む人が、全体構成・各パッケージの責務・ROS2 通信の流れを短時間で把握できるようにすることを目的とする。加えて、リポジトリ内の設計文書（.codex/docs/、各パッケージ内の docs/）と実際のコードとの間に見つかった相違点を明示し、「今どちらが正しいか」を判断できるようにする。

対象読者

  • このリポジトリに初めて触れる開発者・レビュアー
  • ROS 2（topic / service / action）の基本と、Isaac Sim / MoveIt2 の役割をおおまかに知っている人

対象外

旧 PoC 実装は削除済みであり、現行アーキテクチャの対象外のため本書では扱わない。

表記ルール

本書は複数の調査結果を突き合わせて作成しており、コードから直接確認できた事実と、コードからは断定できない解釈を区別して記載する。

事実 コード・設定ファイルを直接読んで確認した内容
推測 コードから断定はできないが妥当と考えられる解釈
乖離 設計意図（旧 ROS2_COMPONENT_ARCHITECTURE.md 等。内容は本書へ統合済みで、原本は削除された）と実装が一致していない箇所

本書全体を通じての注意（2026-07-07 更新） 「単一プロセス構成（robot_node.py / HarvestRuntime）→ マルチノード構成」への移行は完了した。旧構成の残骸（api/・app/・robot_node.py・runtime.py・robot/trajectory_tracking/）は削除済み。さらに 2026-07-07 に robot/ 配下をコンポーネント毎のサブパッケージへ再編し（5章）、各サブパッケージが自前の tests/ を持つ構成になった。加えて同日に motion_command_executor_node は Python パッケージから src/franka_ros2_control/ の C++ ノードへ移設された。本書は実装コードを一次情報として整理している。

2. リポジトリ概要

何を実現するシステムか

事実 README.md 冒頭より：「Isaac Sim 6.0 と ROS 2 Jazzy を1コンテナにまとめ、Franka Panda でトマト収穫シナリオを検証するリポジトリ」。pyproject.toml 上のパッケージ名は tomato-harvest-sim（version 0.1.0, Python ≥3.12）。

カメラで検出したトマトの位置に対して、MoveIt2 で軌道計画を行い、Franka Panda アームで pregrasp → grasp → detach（引き抜き）→ place（トレイへ設置）→ home 復帰、という一連の収穫サイクルを Isaac Sim 上で実行し、その成否を PhysX の物理挙動で判定する。

動作環境

  • Linux + NVIDIA GPU + NVIDIA Container Toolkit + Docker
  • ベースイメージ nvcr.io/nvidia/isaac-sim:6.0.0
  • GUI 表示する場合は xhost +local:root が必要

4.2 IsaacSceneRuntime / physics_harvest（論理状態と物理判定）

事実 scene_runtime.py の IsaacSceneRuntime がシーンの論理状態（phase・tomato状態・gripper開閉・robot pose 等）を一元管理する。physics_grasp_enabled フラグにより「PhysX駆動モード」と「幾何学フォールバック（ソフトウェア判定）モード」を切り替える。

  • 幾何学フォールバックモード（physics_grasp_enabled=False）: ロボットツールとトマトの距離のしきい値（STABLE_GRASP_TOLERANCE_M=0.04m 等）で把持成否をソフトウェア的に判定する。
  • PhysX駆動モード（physics_grasp_enabled=True、--grasp-mode success 時）: close_gripper()/open_gripper() はほぼ何もせず、physics_harvest.py の PhysX 接触判定が正本になる。

physics_harvest.py の IsaacPhysicsHarvestBridge が PhysX の剛体・joint・接触イベントを使い、把持・detach・place・fallをすべて物理ベースで判定する（事実、主要定数：STEM_BREAK_FORCE_N=50, TOMATO_MASS_KG=0.03, DETACH_DISTANCE_M=0.02, PLACE_DISTANCE_M=0.18）。PLACE_DISTANCE_M は、place_pose がトレイ中心の 0.12m 上空（pregrasp_planner の place_vertical_offset_m）でリリースする設計のため「リリース高さ 0.12 + 到達許容 0.05 + 余裕」で 0.18 に設定されている（旧値 0.06 では正常な設置でも必ず FALLEN 判定になる不整合があり、2026-07-06 に修正）。

  1. 接触イベントは subscribe_contact_report_events() でPhysXから非同期に受け取り、一時的な瞬断を吸収するグレース猶予（CONTACT_LATCH_GRACE_STEPS=3）で確定させる。
  2. 接触イベントだけで両指接触が揃わない場合、hand-tomato距離やfinger間ギャップから幾何学的に補完する fallback がある。
  3. gripper が閉じ・両指接触が確定すると hand↔tomato の FixedJoint（grasp joint）を生成 = HELD。
  4. 茎アンカーとの距離が DETACH_DISTANCE_M を超えると DETACHED。
  5. gripperが開き、トレイとの距離が PLACE_DISTANCE_M 以下なら PLACED、そうでなければ FALLEN。

事実 旧版の本書で「配線が外れている疑い」と指摘していた scene_runtime_view.py・debug_visualization.py・isaac_ros2_control_system.py は、どこからも import されていないことを再確認したうえで 2026-07-07 に削除済み（専用テスト2本も同時に削除、README のモジュール構成図も更新済み）。HardwareControlPort プロトコル自体は msg/hardware_control.py に定義があり、ros2_joint_state_hardware_port.py が引き続き使用している。

4.4 control_panel / isaac_viewer（UIと同期処理）

事実 control_panel.py の IsaacControlPanelWindow は Tomato Harvest Controls という3DView上のUIウィンドウで、Start / Stop / Reset / Fixed Camera / Hand Camera ボタンを提供する。ボタン操作は scene_runtime を直接叩くのではなく、/tomato_harvest/control へ publish するワンクッションを置いた設計になっている（同一プロセス内でも ROS2 pub/sub を経由）。

Isaac Sim の物理ステップとの同期には、物理コールバックAPI（subscribe_physics_step_events）は使われていない。代わりに isaac_viewer.py の _step() が simulation_app.update() を挟んで「前処理（コマンド適用）→ 物理ステップ実行 → 後処理（把持判定）」を手続き的に毎フレーム実行する構造になっている。

5.4 trajectory_planner_node と MoveIt2

計画の起点は「phase が TARGET_FOUND になった時」または「trajectory_status が aborted の時」。build_planner()が実運用のMoveIt2ServiceBridgePlannerを構築し、pregrasp → grasp → pull（トマトをattachした軌道） → （pre_placeがあれば）approach → place の順にフルチェーンで計画する。goal の中身・タイミング・切り替えの詳細は 5.5 を参照。

MoveIt2 サービス名が設計文書と異なる乖離
実装（moveit_service_bridge.py）が呼び出すサービス名は既定で /plan_kinematic_path と /apply_planning_scene（環境変数 TOMATO_HARVEST_MOVEIT_SERVICE / TOMATO_HARVEST_MOVEIT_SCENE_SERVICE で変更可）。設計文書が言う /move_group/get_motion_plan とはサービス名が一致しない（メッセージ型自体は GetMotionPlan 相当で同じ）。
フェーズ別の部分再計画は未実装乖離
設計文書は「MOVING_TO_PLACEフェーズでは現在jointからplace軌道のみ再計画する」とするが、trajectory_planner_node.py にフェーズ別分岐は無く、trajectory_status=="aborted" を受けると常にフルチェーンを再計画する。部分再計画用API（plan_place_from_joint_state()）は定義されているが呼ばれていない。

5.5 MoveIt2 の使い方 詳解（goal の内容と切り替えタイミング）

この節では、MoveIt2 を初めて扱う読者向けに、このリポジトリが MoveIt2 に「何を・いつ・どう依頼しているか」と「作られた軌道がいつ差し替わるか」を、実装（motion_planner/moveit_service_bridge.py・motion_planner/node.py・motion_command_executor_node.cpp）に即して説明する。すべて事実（コード確認済み）である。

5.5.1 まず押さえる大前提：MoveIt2 は「計画」だけ、実行は自前

MoveIt2 の一般的な使い方には「MoveGroup アクションで計画も実行も任せる」方法があるが、このリポジトリはそうしていない。MoveIt2 には GetMotionPlan サービス（/plan_kinematic_path）で関節軌道の計算だけを依頼し、返ってきた軌道は一旦 harvest_motion_plan topic に載せて自前のノード群へ渡す。実際にアームを動かす（JTC へ goal を送る）のは C++ の motion_command_executor_node である。つまり登場する「goal」は次の2種類あり、混同しないことが重要:

5.5.2 計画はいつ走るか（計画 goal を与えるタイミング）

trajectory_planner_node が MoveIt2 を呼ぶトリガは次の2つだけ（node.py の _on_phase / _on_trajectory_status）:

  1. phase が TARGET_FOUND になった時 — 収穫サイクル開始時の本計画。target_estimate と /joint_states を両方受信済みであることが前提（未受信なら何もしない）。
  2. trajectory_status が "aborted" になった時 — 実行失敗時の再計画。JTC が abort → executor が execution_status="aborted" → trajectory_monitor_node が trajectory_status="aborted" に変換、という経路で伝わる。

逆に言うと、フェーズが進むたびに計画し直すことはない。1回の計画で収穫サイクル全区間の軌道をまとめて作り、以降は実行側がそれを順番に消費する。

5.5.4 1区間の計画で MoveIt2 に何を送るか

1回の計画は pregrasp → grasp → pull → (pre_place) → place の順に、区間ごとに次の2ステップを繰り返す（_plan_phase()）:

ステップ1: /apply_planning_scene — 衝突環境の構築。MoveIt2 は「どこにぶつかってはいけないか」を知らないので、毎区間の計画前に衝突オブジェクトを送り直す。枝（box 0.18×0.02×0.02）・茎（box）・トレイ（板5枚組）を scene_snapshot の実座標に配置する。ここで重要なのがトマトの扱いの切り替え:

  • pregrasp / grasp 区間（attach_tomato=False）: トマトは「世界に固定された障害物」として置く…のではなく、把持対象なので衝突体としては add しつつ、ハンドが接近できる必要がある。
  • pull / pre_place / place 区間（attach_tomato=True）: トマトを半径 0.01 m の球として panda_hand に attach（ハンド先端から z +0.1034 の位置）する。これにより MoveIt2 は「ハンドがトマトを持ったまま」の状態で、トマトが枝やトレイにぶつからない軌道を計画する。

ステップ2: /plan_kinematic_path（GetMotionPlan） — 計画依頼。MotionPlanRequest の主要フィールドは次のとおり:

フィールド値初心者向け解説
group_namepanda_arm（7軸）SRDF で定義されたアームの関節グループ。finger は計画対象外
start_state最初の区間は現在の /joint_states（URDF 上下限にクランプ済み）。2区間目以降は「前区間の軌道の終端姿勢」後述 5.5.5 の「数珠つなぎ計画」の肝
goal_constraints（位置）目標位置を中心とする半径 0.01 m の球（PositionConstraint）に panda_hand link を入れること「1 cm 以内に届けば成功」という位置ゴール。なお運用上の目標はツール先端なので、panda_hand link 位置へは +0.0584 m の z オフセットで換算する

応答の error_code==1（SUCCESS）なら軌道を受理する。ただし「開始姿勢とほぼ同じ姿勢に留まる no-op 軌道」（全点が許容 1e-3 rad 以内）は棄却して失敗扱いにする — 目標に既に居ると誤認して腕が動かないまま成功になる事故を防ぐため。

5.5.5 数珠つなぎ計画（なぜ start_state を引き継ぐか）

2区間目以降の計画は、ロボットの現在姿勢からではなく、前区間の軌道の終端姿勢から始める（_joint_state_from_trajectory() で終端を取り出して次の start_state に渡す）。理由は単純で、実行時には pregrasp 軌道を走り終えた姿勢から grasp 軌道が始まるからである。もし全区間を現在姿勢から計画すると、区間の継ぎ目で姿勢が飛ぶ軌道になってしまう。

5.5.6 実行 goal の送信と切り替えタイミング

計画結果（最大4本の関節軌道を含む harvest_motion_plan）は1回だけ publish され、以降はフェーズの進行が軌道の切り替えスイッチになる。流れは次のとおり:

  1. behavior_planner_node がフェーズを1段進めて publish する（例: MOVING_TO_PREGRASP → MOVING_TO_GRASP）。
  2. motion_command_node は phase / plan / joint_states のどれかを受信するたびに build_motion_command() を呼び、そのフェーズに対応する軌道1本を載せた motion_command を publish する（対応表は 5.3）。
  3. C++ executor は motion_command を受けるたびに send_trajectory() を実行する。このとき実行中の JTC goal が残っていれば必ず先に async_cancel_goal() で取り消し、新しい FollowJointTrajectory goal を送る。JTC 側の goal は常に最新の1本だけ、が不変条件である。

つまり実行 goal の切り替えは「フェーズ遷移のたび」と「再計画で新しい plan が届いたとき」に起きる。後者では phase は変わらないが、同じフェーズの軌道が新しいものに差し替わって再送される。また AT_GRASP / GRASP_EVALUATION / PLACED では「現在位置に留まる1点だけの停止軌道」が goal として送られる — 把持判定や settle 待ちの間、JTC に明示的に静止を指示するためである。

goal の結果は次のように execution_status へ写像される: 受理拒否または結果 abort → "aborted"、正常完了 → "succeeded"、cancel（こちらから取り消した場合）→ 何も publish しない。cancel を aborted 扱いにすると、フェーズ遷移のたびに偽の失敗が behavior_planner へ流れてしまうからである。

  • ゴール判定許容値（constraints）: 位置ゴール判定は全関節 goal: 0.0 で無効化。stopped_velocity_tolerance: 0.05 は「腕が実際に到着して停止した」ことの確認として機能しており、0.0（無効化）にしてはならない — 無効化すると軌道時間経過だけで即 success となり、腕の到着前にグリッパーが閉じて把持が空振りする（2026-07-07 の実験で確認済み、yaml コメントに記録）。トマト把持中の残留振動による abort は behavior_planner 側の成果ベース遷移（5.2）で救済する。

motion_command_executor_node

事実 2026-07-07 に src/tomato_harvest_sim/robot/motion_command_executor_node/ から src/franka_ros2_control/src/motion_command_executor_node.cpp へ移設済み。/tomato_harvest/motion_command トピックは、現在は「アプリケーション層（robot・Python）→ ロボット制御層（franka_ros2_control・C++）」の境界として実装されている。移行の主眼は性能ではなくレイヤ分離にある:

  • 依存方向の整理: robot 側 Python から control_msgs / FollowJointTrajectory アクションといった ros2_control の知識が消え、アプリ層は「コマンドの意味」だけを、制御層は「実行手段」だけを知る構造になる。
  • sim / 実機切替の局所化: シミュレータ（IsaacSimHardwareInterface）と実機ハードウェアの差し替えが franka_ros2_control 内で完結し、アプリ側は変更不要になる。
  • 責務の固定: motion_command_node は command JSON の生成のみに集中し、JTC action client は C++ executor に閉じ込められる。
方向Topic/Action内容
Sub/tomato_harvest/motion_command（JSON）command_name / gripper_closed / phase_motion_plan.joint_trajectory を YAML parser で読み取り
Action Client/joint_trajectory_controller/follow_joint_trajectory（ROS パラメータ follow_joint_trajectory_action で変更可、2026-07-08 パラメータ化）軌道点の position と time_from_start_sec を JTC action goal へ変換
Pub/tomato_harvest/execution_statusプレーン文字列 "idle"/"running"/"succeeded"/"aborted"
Pub/tomato_harvest/gripper_closed"true"/"false" を publish。SimulatorNode は scene state 用に、IsaacSimHardwareInterface は finger command 用に購読する

移動コマンド（move_* / pull_*）で joint_trajectory が null の場合は、計画失敗とみなして即座に "aborted" を publish し再計画を促す。停止コマンド（hold_*）は軌道 null が正常のため何も publish しない。この判定は command_name の接頭辞で行う。

現状の整理事実
finger 指令の HWI 統合は 2026-07-07 に実装済み。/tomato_harvest/gripper_closed は依然 topic として残るが、これは「scene の論理状態を伝える経路」と「HWI へ finger intent を渡す経路」の兼用であり、simulator の articulation を直接書き換える経路ではない。

launch構成

  • franka_ros2_control.launch.py: ros2_control_node → joint_state_broadcaster spawner → joint_trajectory_controller spawner → motion_command_executor_node の4ノードを起動。controller 名は launch 内の変数で一元管理され、spawner 引数と executor の follow_joint_trajectory_action パラメータの両方がそこから導出される。
  • move_group.launch.py: URDF/SRDF/kinematics/ompl/joint_limitsをロードし move_group ノードを起動（/plan_kinematic_path, /apply_planning_scene を提供）。

その他設定ファイル：joint_limits.yaml（MoveIt2用の速度・加速度上書き）、kinematics.yaml（KDL）、ompl_planning.yaml（RRTConnect等）、panda.srdf（panda_armグループ、自己衝突無効化ペア）。

9. ビルド・実行方法

Dockerビルド

./build.sh
# 環境変数で上書き可能
IMAGE_NAME=my-tomato-sim ISAAC_SIM_IMAGE=nvcr.io/nvidia/isaac-sim:6.0.0 ./build.sh

docker/Dockerfile は nvcr.io/nvidia/isaac-sim:6.0.0 をベースに ROS2 Jazzy（ros-jazzy-desktop, ros-jazzy-moveit, ros2-control系）をインストールし、colcon build --symlink-install で全ROS2パッケージをビルドする。

デバッグコンテナに入る

xhost +local:root
./into.sh

into.sh はコンテナ名 tomato-harvest-sim-debug を作成/再利用し、リポジトリ全体を /workspace/tomato-harvest にマウント、PYTHONPATH=/workspace/tomato-harvest/src を設定する。

コンテナ内での実行

# フルROS2マルチノード構成（推奨）
./scripts/run_ros2_components.sh --isaac --moveit

# 自動開始
./scripts/run_ros2_components.sh --isaac --moveit --auto-start

# headless E2E
./scripts/run_ros2_components.sh --isaac --moveit --headless --headless-steps 600 --auto-start

run_ros2_components.sh の主なオプション

オプション内容
--isaacIsaac Sim統合モード（未指定時はtoy physicsモード）
--headless / --headless-steps Nヘッドレス実行（デフォルト64ステップ）
--auto-start起動後に自動で"start"コマンドを送信
--rebuildC++パッケージを強制再ビルド
--moveitMoveIt2 move_groupを起動
--ros-distro, --ws-dir, --controller-log, --robot-log環境調整用

終了時は trap cleanup EXIT INT TERM により全バックグラウンドプロセスと ros2_control_node/spawner の残留プロセスをkillする。個別に強制終了したい場合は scripts/kill_ros2_nodes.sh（ros2ノード一覧表示→SIGINT→2秒待機→SIGKILL→ros2 daemon stop）を使う。

デバッグ用可視化

scripts/plot_joint_log.py: ログファイルから軌道追従の目標値/実測値、フェーズ遷移、リプラン発生を抽出し、7関節の時系列をフェーズ遷移の色分けマーカー付きでプロットする。
