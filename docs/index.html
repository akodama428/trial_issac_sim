tomato-harvest-sim 解説書

Isaac Sim 6.0 + ROS 2 Jazzy + Franka Panda によるトマト収穫シミュレータのアーキテクチャ解説

1. はじめに

本書の目的

このリポジトリを初めて読む人が、全体構成・各パッケージの責務・ROS2 通信の流れを短時間で把握できるようにすることを目的とする。加えて、リポジトリ内の設計文書（.codex/docs/、各パッケージ内の docs/）と実際のコードとの間に見つかった相違点を明示し、「今どちらが正しいか」を判断できるようにする。

対象読者

  • このリポジトリに初めて触れる開発者・レビュアー
  • ROS 2（topic / service / action）の基本と、Isaac Sim / MoveIt2 の役割をおおまかに知っている人

対象外

poc_code/ ディレクトリは旧 PoC の退避先であり、現行アーキテクチャの対象外のため本書では扱わない。

表記ルール

本書は複数の調査結果を突き合わせて作成しており、コードから直接確認できた事実と、コードからは断定できない解釈を区別して記載する。

事実 コード・設定ファイルを直接読んで確認した内容
推測 コードから断定はできないが妥当と考えられる解釈
乖離 設計意図（旧 ROS2_COMPONENT_ARCHITECTURE.md 等。内容は本書へ統合済みで、原本は削除された）と実装が一致していない箇所

本書全体を通じての注意（2026-07-07 更新） 「単一プロセス構成（robot_node.py / HarvestRuntime）→ 6ノードの ROS2 マルチノード構成」への移行は完了した。旧構成の残骸（api/・app/・robot_node.py・runtime.py・robot/trajectory_tracking/）は削除済み。さらに 2026-07-07 に robot/ 配下をコンポーネント毎のサブパッケージへ再編し（5章）、各サブパッケージが自前の tests/ を持つ構成になった。設計文書 robot/docs/ は本書へ統合して削除した。本書は実装コードを一次情報として整理している。

2. リポジトリ概要

何を実現するシステムか

事実 README.md 冒頭より：「Isaac Sim 6.0 と ROS 2 Jazzy を1コンテナにまとめ、Franka Panda でトマト収穫シナリオを検証するリポジトリ」。pyproject.toml 上のパッケージ名は tomato-harvest-sim（version 0.1.0, Python ≥3.12）。

カメラで検出したトマトの位置に対して、MoveIt2 で軌道計画を行い、Franka Panda アームで pregrasp → grasp → detach（引き抜き）→ place（トレイへ設置）→ home 復帰、という一連の収穫サイクルを Isaac Sim 上で実行し、その成否を PhysX の物理挙動で判定する。

動作環境

  • Linux + NVIDIA GPU + NVIDIA Container Toolkit + Docker
  • ベースイメージ nvcr.io/nvidia/isaac-sim:6.0.0
  • GUI 表示する場合は xhost +local:root が必要

3. 全体アーキテクチャ

事実 実行時は 1 コンテナ内で複数プロセスが ROS 2 topic / service / action 経由で協調動作する。scripts/run_ros2_components.sh が起動する構成を正とすると、実体は次の4グループに分かれる。

最重要ポイント：グリッパー制御は C++ 側を一切経由しない franka_ros2_control の URDF（franka_ros2_control.urdf）にはフィンガー関節（panda_finger_joint1/2）が定義されておらず、IsaacSimHardwareInterface はアーム7軸の command/state interface しか持たない。事実
実際のグリッパー開閉は motion_command_executor_node.py が /tomato_harvest/gripper_closed（std_msgs/String、"true"/"false"）を独立 publish し、これを simulator 側の IsaacJointRos2Bridge が直接 subscribe してフィンガー関節を Isaac Sim の articulation へ直接書き込む、という「robotノード → simulatorノード」の直接配線で実現している。設計文書 ROS2_COMPONENT_ARCHITECTURE.md が記す「C++ MotionCommandExecutor が finger 指令を HWI 経由で送る」という構成とは異なる。乖離

起動経路が2系統存在する

乖離 このリポジトリには実質2つの起動経路があり、完全には同期していないことが確認できた。

  • 系統A（scripts/run_ros2_components.sh）: 上図の4グループをフルに起動する、現時点で最も完成度の高いマルチノード構成。詳細は9章。
  • 系統B（README推奨の scripts/run_harvest_viewer.py --transport ros2 単独起動）: Isaac Sim 内で SimulatorNode と IsaacJointRos2Bridge のみを起動する。scripts/BOOT_SEQUENCE.md はこの経路について「viewer プロセス内で HarvestRuntime まで組み上げる」という旧アーキテクチャの記述が残っており、現行の isaac_viewer.py（HarvestRuntime を持たず SimulatorNode のみを起動する、という明示コメントあり）とは一致しない。

本書の4〜8章は、実装として最も筋が通っている系統A（フルマルチノード構成）を基準に解説する。

4.2 IsaacSceneRuntime / physics_harvest（論理状態と物理判定）

事実 scene_runtime.py の IsaacSceneRuntime がシーンの論理状態（phase・tomato状態・gripper開閉・robot pose 等）を一元管理する。physics_grasp_enabled フラグにより「PhysX駆動モード」と「幾何学フォールバック（ソフトウェア判定）モード」を切り替える。

  • 幾何学フォールバックモード（physics_grasp_enabled=False）: ロボットツールとトマトの距離のしきい値（STABLE_GRASP_TOLERANCE_M=0.04m 等）で把持成否をソフトウェア的に判定する。
  • PhysX駆動モード（physics_grasp_enabled=True、--grasp-mode success 時）: close_gripper()/open_gripper() はほぼ何もせず、physics_harvest.py の PhysX 接触判定が正本になる。

physics_harvest.py の IsaacPhysicsHarvestBridge が PhysX の剛体・joint・接触イベントを使い、把持・detach・place・fallをすべて物理ベースで判定する（事実、主要定数：STEM_BREAK_FORCE_N=50, TOMATO_MASS_KG=0.03, DETACH_DISTANCE_M=0.02, PLACE_DISTANCE_M=0.18）。PLACE_DISTANCE_M は、place_pose がトレイ中心の 0.12m 上空（pregrasp_planner の place_vertical_offset_m）でリリースする設計のため「リリース高さ 0.12 + 到達許容 0.05 + 余裕」で 0.18 に設定されている（旧値 0.06 では正常な設置でも必ず FALLEN 判定になる不整合があり、2026-07-06 に修正）。

  1. 接触イベントは subscribe_contact_report_events() でPhysXから非同期に受け取り、一時的な瞬断を吸収するグレース猶予（CONTACT_LATCH_GRACE_STEPS=3）で確定させる。
  2. 接触イベントだけで両指接触が揃わない場合、hand-tomato距離やfinger間ギャップから幾何学的に補完する fallback がある。
  3. gripper が閉じ・両指接触が確定すると hand↔tomato の FixedJoint（grasp joint）を生成 = HELD。
  4. 茎アンカーとの距離が DETACH_DISTANCE_M を超えると DETACHED。
  5. gripperが開き、トレイとの距離が PLACE_DISTANCE_M 以下なら PLACED、そうでなければ FALLEN。
配線が外れている疑いのあるモジュール乖離
scene_runtime_view.py（USD表示反映）、debug_visualization.py（デバッグマーカー計算）、isaac_ros2_control_system.py（HardwareControlPort実装）は、現行の isaac_viewer.py メインループからは import されていないことを確認した。README のモジュール構成図には登場するが、ROS2コンポーネント分離のリファクタリング途中で呼び出し元が失われた可能性が高い（推測）。

4.4 control_panel / isaac_viewer（UIと同期処理）

事実 control_panel.py の IsaacControlPanelWindow は Tomato Harvest Controls という3DView上のUIウィンドウで、Start / Stop / Reset / Fixed Camera / Hand Camera ボタンを提供する。ボタン操作は scene_runtime を直接叩くのではなく、/tomato_harvest/control へ publish するワンクッションを置いた設計になっている（同一プロセス内でも ROS2 pub/sub を経由）。

Isaac Sim の物理ステップとの同期には、物理コールバックAPI（subscribe_physics_step_events）は使われていない。代わりに isaac_viewer.py の _step() が simulation_app.update() を挟んで「前処理（コマンド適用）→ 物理ステップ実行 → 後処理（把持判定）」を手続き的に毎フレーム実行する構造になっている。

5. tomato_harvest_robot（ROS2ノード群）

事実 現行実装は6つの独立した ROS2 Python ノードで構成される（pyproject.toml の console_scripts と一致）。設計上は5ノード+C++実行体という構成だが、実際には6つ目の motion_command_executor_node もPythonノードとして実装済みである（C++ 移行計画については6章参照）。

パッケージ構成（2026-07-07 リファクタ後）

事実 各コンポーネントは robot/ 配下のサブパッケージとして分離されており、ノード実装（node.py）・ライブラリ・単体テスト（tests/）を1パッケージに同居させる。ノードと機能ライブラリが対応するもの（behavior_planner / perception / motion_planner）は同一パッケージへ統合されている。

1回の motion_command 受信につき、goal送信直前に "running"、結果確定後に "succeeded" または "aborted" を publish する（意図的キャンセル時は何も publish しない）。control_msgs が import できない環境では trajectory 送信は無効化されるが、gripper_closed の publish は独立して継続する。

事実 移動コマンド（move_* / pull_*）で joint_trajectory が null の場合は、計画失敗とみなして即座に "aborted" を publish し再計画を促す（2026-07-06 追加。以前は黙って無視していたため、計画失敗時にシステム全体がデッドロックしていた）。停止コマンド（hold_*）は軌道 null が正常のため何も publish しない。この判定は command_name の接頭辞で行う。

launch構成に含まれていない乖離
robot/launch/robot_nodes.launch.py は①〜⑤の5ノードのみを起動し、⑥ motion_command_executor_node を含んでいない。この launch ファイル単体では motion_command を JTC action に変換する主体が起動されず、システムとして完結しない。scripts/run_ros2_components.sh は6ノードすべてを起動するため、実運用ではこちらが使われていると考えられる（推測）。

5.4 trajectory_planner_node と MoveIt2

計画の起点は「phase が TARGET_FOUND になった時」または「trajectory_status が aborted の時」。build_planner()（環境変数 TOMATO_HARVEST_PLANNER_BACKEND で auto/geometric/moveit2 を切替、既定 auto は rclpy/moveit_msgs の import可否で自動選択）で得たプランナーが、pregrasp → grasp → pull（トマトをattachした軌道） → （pre_placeがあれば）approach → place の順にフルチェーンで計画する。

MoveIt2 サービス名が設計文書と異なる乖離
実装（moveit_service_bridge.py）が呼び出すサービス名は既定で /plan_kinematic_path と /apply_planning_scene（環境変数 TOMATO_HARVEST_MOVEIT_SERVICE / TOMATO_HARVEST_MOVEIT_SCENE_SERVICE で変更可）。設計文書が言う /move_group/get_motion_plan とはサービス名が一致しない（メッセージ型自体は GetMotionPlan 相当で同じ）。
フェーズ別の部分再計画は未実装乖離
設計文書は「MOVING_TO_PLACEフェーズでは現在jointからplace軌道のみ再計画する」とするが、trajectory_planner_node.py にフェーズ別分岐は無く、trajectory_status=="aborted" を受けると常にフルチェーンを再計画する。部分再計画用API（plan_place_from_joint_state()）は定義されているが呼ばれていない。

5.5 レガシーコードの削除履歴

事実 旧アーキテクチャの軌道追従ライブラリ robot/trajectory_tracking/（coordinator / tracker / action_client 等）は、現行6ノードからの参照ゼロを確認したうえで 2026-07-07 に削除済み。旧単一プロセス構成の api/・app/・robot_node.py・runtime.py も削除済み。robot/msg/trajectory_tracking.py（型定義モジュール）だけは simulator/isaac_franka_driver.py が参照するため残存している。

6. franka_ros2_control（C++側）

事実 hardware_interface::SystemInterface を継承した IsaacSimHardwareInterface（franka_ros2_control_plugin.xml で pluginlib 登録）が中核。アーム7軸のみを扱い、グリッパー関節は扱わない（3章参照）。

IsaacSimHardwareInterface の入出力
方向Topic内容
Sub/isaac_joint_statessimulator側 IsaacJointRos2Bridge から。名前一致でstateへマッピング
Pub/isaac_joint_commandsURDF上下限でクランプ済みのcommandを送信

起動時の安全策

  • on_activate: 起動後最大5秒 Isaac Sim からの初回 joint_state を待機。届かなければ0埋めで警告を出しつつ継続し、届いた実位置で command をリシードして起動時のジャンプを防止。
  • write(): state_received_ が false の間は何もpublishしない（接続前に0を送るとJTCがゼロホールドしてしまうため、とコメントに明記）。

controller構成（franka_controllers.yaml）

  • controller_manager: update_rate: 100Hz。use_sim_time は意図的に未設定（/clock配信より先にactivateが試行されるため、とコメントあり）。
  • joint_state_broadcaster: /joint_states をpublish。
  • joint_trajectory_controller: panda_joint1〜7、position+velocity interface、interpolation_method: splines、open_loop_control: true（closed-loopだと panda_joint4 が可動域外の0へ動く問題を回避するためのコメントあり）。
  • ゴール判定許容値（constraints）: 位置ゴール判定は全関節 goal: 0.0 で無効化。stopped_velocity_tolerance: 0.05 は「腕が実際に到着して停止した」ことの確認として機能しており、0.0（無効化）にしてはならない — 無効化すると軌道時間経過だけで即 success となり、腕の到着前にグリッパーが閉じて把持が空振りする（2026-07-07 の実験で確認済み、yaml コメントに記録）。トマト把持中の残留振動による abort は behavior_planner 側の成果ベース遷移（5.2）で救済する。

将来計画: MotionCommandExecutor の C++ 移行

推測 現在 Python で実装されている motion_command_executor_node（5.1⑥）は、設計上は franka_ros2_control 内の C++ ノード MotionCommandExecutor（未実装）へ置き換える計画である。/tomato_harvest/motion_command トピックはその際の「アプリケーション層（robot・Python）→ ロボット制御層（franka_ros2_control・C++）」の境界として設計されている。移行のメリットは性能ではなくレイヤ分離にある:

  • 依存方向の整理: robot 側 Python から control_msgs / FollowJointTrajectory アクションといった ros2_control の知識が消え、アプリ層は「コマンドの意味」だけを、制御層は「実行手段」だけを知る構造になる。
  • sim / 実機切替の局所化: シミュレータ（IsaacSimHardwareInterface）と実機ハードウェアの差し替えが franka_ros2_control 内で完結し、アプリ側は変更不要になる。
  • gripper 制御の正規化: 現在の「robot ノード → simulator への直接配線」（3章の乖離参照）を、HWI 経由の finger 指令に一本化できる。

この計画がある限り、motion_command_node と motion_command_executor_node を統合してはならない（境界そのものが消えるため）。計画を放棄する場合は先に本書のこの節を更新すること。

launch構成

  • franka_ros2_control.launch.py: ros2_control_node → joint_state_broadcaster spawner → joint_trajectory_controller spawner の3ノードを起動。
  • move_group.launch.py: URDF/SRDF/kinematics/ompl/joint_limitsをロードし move_group ノードを起動（/plan_kinematic_path, /apply_planning_scene を提供）。

その他設定ファイル：joint_limits.yaml（MoveIt2用の速度・加速度上書き）、kinematics.yaml（KDL）、ompl_planning.yaml（RRTConnect等）、panda.srdf（panda_armグループ、自己衝突無効化ペア）。
