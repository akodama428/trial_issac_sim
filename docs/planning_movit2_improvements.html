Planning Draft

MoveIt2 リプラン改善計画

現在の trajectory_planner_node は、TARGET_FOUND と trajectory_status="aborted" を契機にしか MoveIt2 を再実行しない。 これは静的な Sense-Plan-Act に近く、制御誤差や外乱がある環境では保守的すぎる。 本書では、「現在状態からの低周期リプランを常時回すべきか」という問いに対して、 その方向性自体は妥当だが、毎回フルチェーンを無条件に再計画する方式は採らない、 という前提で段階的な改善計画を定義する。

更新日: 2026-07-09 / 対象: src/tomato_harvest_sim/robot/motion_planner/, src/franka_ros2_control/, docs/index.html

1. 現状認識

  • 現行実装では、計画トリガは phase == TARGET_FOUND と trajectory_status == "aborted" の2つだけである。
  • 1回の planning で pregrasp → grasp → pull → (pre_place) → place のフルチェーンを作り、その後は executor が順に消費する。
  • MOVING_TO_PLACE 向けの plan_place_trajectory() は既に実装済みだが、現状は呼ばれていない。
  • executor は新しい trajectory を受けるたびに、既存 goal を cancel して FollowJointTrajectory goal を差し替える。
問題は「replan が少なすぎる」ことだけではない。現行のまま timer を足して毎秒フルチェーン再計画すると、 JTC cancel → 新 goal 送信 が高頻度で繰り返され、接触中フェーズや終端近傍で不安定化する。

2. 一次情報から見た設計判断

MoveIt 公式の Hybrid Planning は、従来の Sense-Plan-Act が動的・不確実環境に弱いことを前提に、 遅い global planner と速い local planner を並列で回す構成を推奨している。 また PlanningSceneMonitor は最新の robot state と planning scene を維持する推奨インタフェースであり、 Realtime Servo は visual servoing や closed-loop position control を対象にしている。

判断: 「現在状態からの再計画を定期実行する」という方向は正しい。
ただし当面の実装方針は、低周期の phase-scoped suffix replan を追加し、 将来的に Hybrid Planning / Servo を使う local planner 併用構成 へ進める。
重要なのは、改善案Bを最終形として固定しないことだ。Bは「低周期replanを足す案」ではなく、 global planning と execution の間に、replan trigger / state aggregation / plan revision / plan adoption policy を先に切り出す段階 として設計する必要がある。この切り出しができていれば、後から local planner を追加しても下流の executor 契約を大きく崩さずに済む。

3. 目標と非目標

目標

  • 制御誤差、残留振動、外乱、scene 更新があっても、active phase の実行継続性を高める。
  • 最新の /joint_states と scene_snapshot を起点に、古い trajectory を放置しない。
  • 再計画が executor の goal 差し替えループを破壊しないよう、周波数・採用条件・phase 境界を明示する。

非目標

  • 初回から full Hybrid Planning を丸ごと導入すること。
  • DETACHING の接触制御を OMPL のグローバル再計画だけで解決すること。
  • 全 phase を同一アルゴリズムで扱うこと。

4. 比較する改善案

案 内容 利点 欠点 判断

5. 推奨アーキテクチャ

近短期は案Bを採る。ただし実装の目的は suffix replan そのものではなく、 「current state を集約し、replan を起動し、候補planを評価し、executorへ採用する」境界を明示的に作ることに置く。 そのうえで、初期の plan producer は low-rate OMPL suffix planner とし、将来はここに local planner を追加して C へ移行する。

モジュール 追加・変更責務
trajectory_planner_node timer 起動、replan 条件判定、phase-scoped suffix planning、plan revision 付与。将来は global planner coordinator へ昇格
moveit_service_bridge.py phase 別 suffix API を明確化。既存 plan_place_trajectory() の活用。必要なら plan_from_phase() を追加。将来は global planner adapter になる
motion_command_node 古い revision の plan を無視。現 phase と plan revision の整合を保持。将来は local planner 由来 command との arbitration 窓口候補
motion_command_executor_node 新 plan が有効採用された場合のみ cancel-and-replace。差分が小さい場合は据え置きも検討。入力契約はなるべく固定し、planner 側進化を吸収しない
trajectory_monitor_node aborted だけでなく、将来的には tracking error や scene change を replan reason として出せるよう拡張候補。Cでは event source の一つになる

6. BからCへつなぐ設計原則

Bで先に作るもの Cでどう使うか Bの段階でやってはいけないこと
State Aggregation
最新 joint_states、scene_snapshot、phase、tracking error をまとめる入力境界		 global planner と local planner の共通入力になる planner実装の中に subscribe と状態整形を埋め込んで密結合にすること
Replan Trigger Policy
timer、abort、scene change、tracking error の起動条件		 Hybrid Planning の event logic / local planner 介入条件へ発展できる aborted 固定の if 文を各ノードへ散らすこと
Plan Revision / Adoption Policy
どの候補planを採用するかの規則		 global plan と local correction command の arbitration に再利用できる 新planが来たら無条件で cancel-and-replace すること
Phase-scoped Plan API
plan_from_phase() のような残区間計画API		 global planner を full-chain から phase-aware planner に差し替えやすくなる phaseごとの分岐を node 側の巨大 if で持つこと
Executor-facing Command Contract
executor に渡す command / trajectory 契約		 planner の進化に対して executor を安定化できる local planner 導入時に executor のI/Fまで同時に壊すこと
BがCにつながる条件は、suffix replan の頻度を上げることではなく、planner の責務を producer / trigger / adoption に分解することである。 この分解ができていなければ、Cへ進むときに B の実装をほぼ捨てることになる。

7. Replan の基本ルール

7.1 周期

  • 初期値は 1.0 Hz ではなく 0.5 Hz から始める。まず controller churn を抑える。
  • 計画時間上限は replan 周期より十分短くする。初期案は allowed_planning_time = 0.20 - 0.35 sec。
  • planner がまだ実行中なら次周期をスキップし、同時多重 plan は許可しない。

7.2 実行条件

  • active phase が移動系であること。
  • 最新 joint_states と scene_snapshot が揃っていること。
  • 前回 replan から最小間隔を超えていること。
  • start state の偏差、target/tray pose の変化、scene object 更新、tracking error のいずれかが閾値を超えた場合だけ採用する。

7.3 採用条件

  • 新 trajectory の先頭が現在関節位置から不連続すぎないこと。
  • 残り実行時間がごく短い場合は差し替えない。
  • 新旧差分が閾値未満なら cancel せず現行 goal を継続する。
  • plan message に plan_revision と planned_phase を持たせ、stale overwrite を防ぐ。

8. Phase 別方針

Phase replan 方針 理由
MOVING_TO_PREGRASP 周期 replan 有効。suffix は pregrasp 以降 自由空間が多く、再計画の利得が大きい
MOVING_TO_GRASP 周期 replan 有効。suffix は grasp 以降 トマト認識誤差の吸収余地がある
DETACHING 初期段階では周期 OMPL replan を原則無効。abort/明示要求時だけ 接触中に global trajectory を差し替えると不安定になりやすい。将来は Servo / local planner 候補
MOVING_TO_PLACE 周期 replan 有効。plan_place_trajectory() を使って現在状態から place suffix のみ計画 既存 API があり、全チェーン再生成より自然
RETURNING_HOME replan 対象外 MoveIt2 非経由の direct 軌道

9. Cへ至る実装ステップ

  1. Step 0: 観測追加
    現行 planner latency、cancel 回数、phase ごとの abort 率、trajectory 差し替え回数をログ化する。これは B でも C でも必要な基準線になる。
  2. Step 1: 契約を安定化
    HarvestMotionPlan かそのシリアライズに plan_revision、generated_at、planned_from_phase、将来の producer_kind を追加する。ここで plan adoption の土台を作る。
  3. Step 2: State Aggregation と Trigger Policy を独立化
    trajectory_planner_node に periodic callback を追加する前に、phase、joint state、scene snapshot、tracking error を一箇所で束ねる。replan 条件も独立した policy として切り出す。
  4. Step 3: Bの最小完成形
    現在 phase に応じて full-chain ではなく残り区間だけ計画する suffix API へ切り替える。まずは MOVING_TO_PLACE を対象にして、executor churn 抑制と revision 採用規則を入れる。
  5. Step 4: Bを一般化
    MOVING_TO_PREGRASP、MOVING_TO_GRASP へ広げる。ここで作るのは「suffix planner」そのものより、phase-aware global planner adapter である。
  6. Step 5: Cの受け皿を追加
    executor の下流契約は変えずに、plan producer を複数持てる形にする。少なくとも global_planner と将来の local_planner を識別できるようにする。
  7. Step 6: local planner を追加
    接触や終端補正が支配的な phase に対し、MoveIt Servo もしくは Hybrid Planning の local planner を導入する。最初は DETACHING ではなく、自由空間末端の微修正区間から始める。
  8. Step 7: Cへ移行
    global planner は疎な full/suffix replan、local planner は高頻度補正、trigger policy は event-based、plan adoption は arbitration 付き、という最終形へ移す。ここで初めて改善案Cが成立する。
つまり B は「低周期OMPL再計画を増やして終わり」ではない。Cに必要な境界を先に作るための移行ステップとしてのみ採用する。

10. テスト計画

  • unit: replan timer の gating、minimum interval、revision 比較、phase 別 suffix 選択。
  • unit: replan trigger policy と plan adoption policy が planner 実装から独立に検証できること。
  • integration: MOVING_TO_PLACE 中に joint state をずらし、place suffix だけが再計画されること。
  • integration: planner 実行中に次周期が来ても二重起動しないこと。
  • integration: 将来の local planner ダミー実装を挿したときも executor 側契約を変えずに command 採用できること。
  • e2e: Isaac + MoveIt + ros2_control 構成で、外乱注入時に abort 頼みではなく trajectory 更新が走ること。
  • regression: DETACHING の成果ベース遷移と JTC abort 救済が壊れないこと。

11. リスク

  • cancel churn: 周期 replan が高すぎると、controller が常に goal 差し替え中になり進捗が落ちる。
  • OMPL 非決定性: 毎回別経路になり、末端で jitter が増える可能性がある。
  • 接触区間との相性: detaching は local correction 向きであり、global replan を多用すると逆効果になりうる。
  • stale overwrite: phase が進んだ後に古い plan が到着すると、意図しない巻き戻りが起こる。
  • Bの作り捨て化: trigger / adoption / state aggregation を切り出さずにBを実装すると、Cへ進む時に全面作り直しになる。

12. 最終提案

最終目標を改善案Cに置くなら、改善案Bは「一時的な低周期replan」ではなく、 Cに必要な責務分離を先行実装するステップとして定義すべきである。
実施順は、 State Aggregation / Trigger Policy / Plan Revision / Adoption Policy の切り出し → phase-scoped suffix replan の導入 → plan producer の複線化 → Hybrid Planning / Servo の local planner 追加 とする。これなら B の成果物を捨てずに C へ進める。
