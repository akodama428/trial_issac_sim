変更計画: トマト収穫シミュの物理忠実度レベルアップ

作成日: 2026-07-08 ／ 状態: draft（実装前レビュー用） ／ 対象: simulator/physics_harvest.py を中心とした把持・分離・設置判定の物理化
本計画は「トマトの枝からの分離（DETACH）と finger による把持を、人工拘束・幾何しきい値に頼らず物理状態のみで再現する」ための段階的変更計画である。実装はまだ行っていない。

1. 目的とスコープ

目的: 現在の収穫サイクルは「把持 = FixedJoint の動的生成」「分離・設置 = 距離しきい値による判定」という疑似物理で成立している。これを PhysX の摩擦・関節破断・自由落下といった物理現象そのもので置き換え、状態判定（HELD / DETACHED / PLACED / FALLEN）は物理の観測に徹する構成へ引き上げる。

スコープ:

  • 対象: physics_harvest.py（判定と拘束）、finger 駆動の力制御化、トマト・finger・トレイの物理マテリアル、シーン物理設定。
  • 対象外: MoveIt2 計画側（planning scene の attached tomato は計画用の近似であり、そのまま維持）、ROS2 ノード構成、C++ 制御層。behavior_planner のフェーズ判定条件は原則変えない（tomato_status の意味は維持し、判定の実現方法だけを差し替える）。
  • 互換性: 既存モード（--grasp-mode success/failure）は温存し、新方式は --grasp-mode physics（新設）のフラグで切り替える。E2E が不安定になった場合は即座に旧モードへ戻せる。

2. 現状分析（As-Is）— 何が幾何・人工で、何が物理か

結論から言うと、「幾何条件のみ」ではないが「物理のみ」でもないハイブリッドである。接触の検出には PhysX の物理イベントを使っているが、把持の保持は人工拘束（FixedJoint）で、分離・設置の判定は距離しきい値の幾何条件である。以下すべて現行コード（physics_harvest.py）で確認した事実。

3. 目標状態（To-Be）

ゴール定義: --grasp-mode physics において、①〜④がすべて成立すること。
① 把持: grasp joint を生成せず、finger の押付け力とトマト表面の摩擦だけでトマトを保持・搬送できる。
② 分離: pull 動作の張力が stem joint の破断しきい値を超えたときにPhysX の関節破断で自然に枝から離れる。
③ 設置/落下: リリース後のトマトの実際の落下・静止位置から PLACED / FALLEN を判定する。
④ 判定は観測のみ: 幾何フォールバック・テレポート復元・リリース瞬間の距離即決を physics モードでは使わない。状態判定コードは物理に一切介入しない（読み取り専用）。

Step 0: 観測基盤とベースライン確保

物理化の各ステップは「数値で良否を語れること」が前提になる。まず現行 success モードの E2E をベースラインとして固定し、チューニングに必要な観測量をログへ出す。

変更内容① contact report のヘッダだけでなく接触力（力積/法線力）を読み取ってデバッグログに出力（現行は第2引数の contact_data を捨てている）。② トマトの速度・hand との相対変位・stem joint への負荷推定を毎ステップログ化。③ 現行 E2E（run15 相当）を 5 回実行し、成功率・各フェーズ所要ステップ数を記録して docs/ に残す。
対象ファイルphysics_harvest.py（ログ追加のみ）、scripts/plot_joint_log.py（必要なら接触力プロット追加）
テスト / 良品条件既存テスト 88 件 + C++ 5 件が全通過（挙動変更ゼロ）。デバッグログに接触力と相対変位が数値で出力される。ベースライン成功率が記録される。
リスク / 戻し方低リスク（ログのみ）。git revert で即戻せる。

Step 1: 物理モデルの土台整備（マテリアル・コリジョン・ソルバ）

変更内容① UsdPhysics.MaterialAPI でトマト表面（高摩擦: 静摩擦 ~1.2 / 動摩擦 ~1.0 / 反発 ~0.1 を初期値に）、finger パッド（ゴム相当）、トレイ・地面のマテリアルを定義し、摩擦合成モードを設定。② トマトのコリジョンを解析球（半径は scene.yaml の tomato_radius_m=0.01 と一致）で明示し、contact offset / rest offset を小径物体向けに調整。③ physics scene のソルバ反復回数（position/velocity iteration）と物理 dt/substeps を小物体把持向けに引き上げ。④ これらのパラメータはハードコードせず config/scene.yaml に新設する physics: セクションへ集約（前回の教訓: 設定はコードに埋めない）。
対象ファイルphysics_harvest.py（_define_tomato_physics() 拡張）、config/scene.yaml / scene_config.py（物理パラメータ追加）、isaac_viewer.py（physics scene 設定）
テスト / 良品条件単体: scene_config が新パラメータを読めること。統合（headless）: ATTACHED 状態で 500 ステップ放置してもテレポート復元（_should_restore_attached_tomato_pose）の発動が 0 回（= 物理が発散しない土台の証明）。既存 E2E success モードの成功率がベースラインから劣化しない。
リスク / 戻し方ソルバ設定変更が既存挙動（アーム追従）へ波及する可能性 → physics モード時のみ適用する分岐も検討。yaml の physics セクションを外せば従来挙動。

Step 2: finger 駆動の力制御化（把持力の上限設定）

変更内容finger 2 関節の drive パラメータ（stiffness / damping / maxForce）を明示設定する。閉指令の位置目標は 0.0 のまま（HWI・C++ 側は無変更）、maxForce で押付け力を数 N 程度に制限し「目標には届かないがトマト径 (~0.02m ギャップ) で力平衡して止まる」状態を作る。適正値は Step 0 のログを見ながらチューニングし、scene.yaml の physics セクションで管理する。
対象ファイルisaac_franka_driver.py または isaac_viewer.py（articulation 初期化時に finger drive gain を設定）、config/scene.yaml
テスト / 良品条件統合（headless）: トマト直上で閉指令 → ①トマトの速度スパイクが発生しない（弾き飛ばさない）、②finger ギャップがトマト径近傍で静定する、③接触力ログが maxForce 近傍で飽和する。success モード E2E は引き続き劣化なし（grasp joint 併用のままなので挙動互換）。
リスク / 戻し方力が弱すぎると Step 3 で滑る、強すぎると弾く — ここが最重要チューニングポイント。drive 設定を外せば USD 既定値に戻る。

Step 3: 摩擦による保持 — grasp joint の廃止（--grasp-mode physics 新設）

変更内容① isaac_viewer.py の --grasp-mode に physics を追加（success / failure は不変）。② physics モードでは _create_grasp_joint() を呼ばず、幾何フォールバック（_augment_contacts_from_grasp_geometry）とテレポート復元も無効化。③ HELD 判定を観測ベースへ: 「gripper 閉 かつ 両指の接触（力積 > しきい値）が N ステップ連続 かつ hand-トマト相対速度 < しきい値」で HELD を報告。以降も相対変位を監視し、滑落（相対距離が把持時から一定以上増加）したら HELD を取り消して FALLEN 系へ。④ 判定パラメータ（N・しきい値）は scene.yaml へ。
対象ファイルphysics_harvest.py（モード分岐 + 観測判定の新設。既存クラスに if を足すのではなく、判定ストラテジを success 用 / physics 用に分離して単一責務を保つ）、isaac_viewer.py、simulator/tests/
テスト / 良品条件単体: 観測判定ロジック（接触継続カウント・相対速度・滑落検出）を pose/速度スタブで検証（test_physics_grasp_runtime.py と同形式）。統合: 把持 → 0.1m 持ち上げ → 5 秒保持でトマトの hand 相対変位 < 5mm（滑落なし）、リリースで自然落下すること。
リスク / 戻し方本計画の最難関。残留振動（G6）で滑る場合は Step 2 の把持力・摩擦係数へ差し戻してチューニング。physics モードが不成立でも success モードは無傷なので、フラグを戻すだけで運用継続できる。

Step 4: 枝の自然分離 — 破断の物理化と観測

変更内容① STEM_BREAK_FORCE_N を現実的な茎の強度（初期候補 2〜5N: トマト自重 0.3N の 7〜17 倍。pull 軌道の張力で切れ、把持振動では切れない水準）へ再設計し scene.yaml へ移動。② DETACHED 判定を「stem joint が実際に破断したか」の検出に変更する。検出方法は 9章の要調査（PhysX の破断イベント購読 or joint の enabled/prim 状態のポーリング）の結果で確定する。③ 距離判定（DETACH_DISTANCE_M）は physics モードでは判定に使わず、デバッグログの参考値に格下げ。
対象ファイルphysics_harvest.py、config/scene.yaml
テスト / 良品条件統合: ①pull 軌道の実行で破断が発生し DETACHED が報告される、②把持だけして引かない場合は 10 秒経過しても破断しない、③破断後にトマトが hand 側に残る（摩擦保持が分離の衝撃に耐える）。
リスク / 戻し方破断しきい値と把持力のバランスが崩れると「切れる前に滑る」。Step 0 の張力ログで pull 時張力 > 破断値 > 振動時張力 の分離余裕を確認してから値を決める。しきい値を 50N に戻せば従来同等。

Step 5: 設置/落下の物理判定 — リリース後の settle 監視

変更内容① リリース（gripper 開）を「判定の瞬間」ではなく「監視の開始」に変更。② リリース後、トマトの実位置と速度を毎ステップ追跡し、静止（速度 < しきい値が N ステップ継続）した時点で判定: 静止位置がトレイ内部領域（base 上面 × 壁内側、scene.yaml のトレイ寸法から算出）なら PLACED、それ以外（地面など）なら FALLEN。③ タイムアウト（例 5 秒）で静止しない場合は FALLEN。④ PLACE_DISTANCE_M による瞬時判定は physics モードでは廃止。
対象ファイルphysics_harvest.py、必要に応じて behavior_planner/node.py の PLACED 待ちタイムアウト確認（現行は tomato_status==PLACED を待つ設計なので原則無変更で済む見込み）
テスト / 良品条件単体: トレイ内部領域判定を座標スタブで検証（内側/壁上/外側/境界）。統合: ①正常リリースで落下 → トレイ内静止 → PLACED、②トレイ外でリリース → FALLEN、③判定確定までの遅延が behavior_planner のタイムアウトに収まる。
リスク / 戻し方バウンドしてトレイ外へ転がり出るフレークが想定される → トレイ壁高さ・反発係数（Step 1）で抑制。判定を瞬時方式に戻すフラグは残さない（physics モード自体を戻せば十分）。

Step 6: 統合検証・チューニング・既定化判断

変更内容① physics モードで E2E（detecting → complete）を 10 回連続実行し、成功率・フレーク要因を記録。② パラメータ（摩擦・把持力・破断値・settle しきい値）の最終値を scene.yaml に確定。③ 成功率が受け入れ条件（10章）を満たしたら、--grasp-mode の既定値を physics へ切り替えるかを判断（切り替えは別コミット）。④ docs/index.html（4.2節）と docs/test_list.html を新モードの仕様・テストで更新。
テスト / 良品条件E2E 成功率 8/10 以上（暫定目標）。既存 success モードの全テスト・E2E が引き続き通過。

調査方法: NVIDIA 公式ドキュメント・公式フォーラム・公式 GitHub の Web 調査に加え、実行環境そのもの（docker コンテナ tomato-harvest-sim-debug 内の Isaac Sim インストール、omni.physx 110.1.11 / omni.usd.schema.physx 110.1.11 の Python バインディング定義ファイル）を一次情報として直接検証した。確認日: 2026-07-08。事実と推測を分離して記載する。

R1: 関節破断の検出方法 — 確定

確認できた事実: コンテナ内 Isaac Sim の omni/physx/bindings/_physx.pyi（omni.physx 110.1.11）に、シミュレーションイベント列挙 SimulationEvent.JOINT_BREAK（値 7）が定義されている。イベント payload は辞書で、'jointPath': int2（破断した joint の USD パスを int 2 個にエンコードしたもの。PhysicsSchemaTools.decodeSdfPath で SdfPath に復元できる）を含む。イベントは get_physx_interface().get_simulation_event_stream_v2()（carb.events.IEventStream を返す）の購読で受け取れる。破断のしきい値は現行コードでも使用中の UsdPhysics.Joint の BreakForce / BreakTorque 属性であり、PhysX 公式ドキュメントでも breakable joints として仕様化されている。

推奨方針: Step 4 の DETACHED 判定は SimulationEvent.JOINT_BREAK の購読で実装する（ポーリング代替は不要）。受信時に jointPath をデコードし、stem joint のパスと一致した場合のみ DETACHED を確定する。既存の contact report 購読（subscribe_contact_report_events）と同居できる。

R2: 接触力（力積）の取得 — 確定

確認できた事実: 現行コードが使用中の subscribe_contact_report_events のコールバック型は Callable[[ContactEventHeaderVector, ContactDataVector], None]。ContactEventHeader は actor0 / actor1 / collider0 / collider1 / type（CONTACT_FOUND / PERSISTS / LOST）に加えて contact_data_offset と num_contact_data を持ち、第2引数の ContactDataVector の該当区間を参照することで接触点ごとの impulse（Float3）・normal・position・separation が取得できる（バインディング定義で全フィールド確認）。現行実装はこの第2引数を捨てており、Step 0 の「接触力ログ」は既存購読の拡張だけで実現できる。報告最小しきい値は使用中の PhysxContactReportAPI.CreateThresholdAttr(0.0) で制御する。

推奨方針: 把持力は「接触力積のノルム ÷ 物理 dt」で法線力相当に換算してログ化する。HELD の観測判定（Step 3）は力積が連続 N ステップしきい値超え、を一次条件にする。

R3: 球体把持のねじり摩擦 — 確定

確認できた事実: PhysxCollisionAPI（collider prim 単位）に physxCollision:torsionalPatchRadius / physxCollision:minTorsionalPatchRadius 属性があり、点接触でもねじり方向の摩擦を発生させる接触パッチ半径を設定できる（スキーマ定義で確認）。同 API には Step 1 で予定していた contactOffset / restOffset も揃っている。またマテリアル側（PhysxMaterialAPI）には frictionCombineMode / restitutionCombineMode に加えて、compliant contact（physxMaterial:compliantContactStiffness / compliantContactDamping）が存在し、剛体のまま「柔らかい接触」を近似できる。

推奨方針: トマト collider（半径 0.01m の球）に torsionalPatchRadius を数 mm（初期値 0.003〜0.005m、要チューニング）で設定する。トマトの実物の柔らかさ（つぶれによる接触面積拡大）の近似として compliant contact の採用を Step 1 のチューニング候補に追加する。推測 具体値は一次情報に規定がないため Step 0 の観測で決める。

R4: finger drive の maxForce — 確定

確認できた事実: OpenUSD 公式 API リファレンスにより、UsdPhysics.DriveAPI は multipleApply スキーマ（prismatic joint には "linear"、revolute には "angular" で適用）で、drive は「力制限付きダンパー付きバネ」として仕様化されている: force = stiffness × (targetPosition − position) + damping × (targetVelocity − velocity)、maxForce がその上限（inf = 無制限）。つまり HWI からの位置指令（drive の targetPosition）はそのままに、maxForce だけで押付け力を制限する Step 2 の設計は USD の仕様どおり成立する。

推奨方針: finger 2 関節（prismatic）の drive:linear:physics:maxForce と stiffness / damping を scene.yaml から設定する。Franka 実機のグリッパー連続把持力が最大 70N（公式仕様）であることを参考上限とし、トマト用初期値は数 N から始める。

R5: Franka 公式 USD アセットの finger コリジョン — 部分確認

確認できた事実: 使用アセットは Isaac/Robots/FrankaRobotics/FrankaPanda/franka.usd（isaac_viewer.py で確認）。コリジョン形状の公式仕様記述は見つからなかったが、関連する既知報告として次を確認した: ① Isaac Lab 公式 GitHub で「Panda グリッパーの collider が視覚形状にタイトフィットしない」issue（#3571）、② franka_alt_fingers.usd で把持時に指が対象へめり込む報告（Isaac Sim 4.1、公式フォーラム）、③ 把持中に finger joint が外れる報告（公式フォーラム 2025）。また Isaac Sim には把持姿勢を検証する公式ツール Grasp Editor が存在する。

推奨方針: 「既定のまま使える」とは仮定しない。Step 1 に「コンテナ内で franka.usd の finger collider（形状・近似方式・マテリアル割当）をスクリプトで検査し、フィット不良ならフィンガーパッドに解析形状（box）の collider を追加する」タスクを組み込む（4章 G1 の具体化）。

R6: グレーゾーン介入の許容方針 — 人間判断待ち

判断材料として確認した事実: NVIDIA 公式フォーラムの安定把持スレッド（Isaac Sim 5.1.0、2025-12〜2026-01）で、NVIDIA スタッフが「物理 timestep の削減と摩擦の増加」を推奨しつつ、「チューニング済みシナリオの一般化は難しく、パラメータが変わるたびに再チューニングが必要」と公式に認めている。質問者は摩擦 10.0 等の非現実値でも改善が限定的だったと報告している。また Isaac Lab には吸着型の人工拘束グリッパー（SurfaceGripper）が公式提供されており、「完全な摩擦物理のみ」に固執しない設計が公式エコシステムでも一般的である。

示唆: 本計画の「摩擦のみで保持」は挑戦的な目標であることが裏付けられた。Step 3 が難航した場合の現実的な妥協ライン（例: 把持中の締め増し、compliant contact の積極利用）をどこに置くかは、M2 マイルストーン評価時に人間が判断する。
