# Implementation

## 目的
AI および開発者が、一貫した品質でソフトウェアを実装するための共通原則と実装手順を定義する。

このスキルはプロジェクト固有の仕様を追加しない。実装時は必ず対象プロジェクトの仕様書と設計書を正本として参照する。

## 先に読むもの
- `AGENTS.md`
- `docs/MASTER.md`
- 実装対象に対応する仕様文書
  - 例: `docs/PROJECT.md`, `docs/ADR.md`, `docs/ARCHITECTURE.md`, `docs/PATTERNS.md`, `docs/TESTING.md`
- 関連する既存コード
- 関連する既存テスト

仕様や設計が不足している場合は、推測で実装しない。不明点は質問するか、課題として明示する。

## 基本方針

### 1. 仕様を正とする
- コードは仕様を実現する手段である。
- 実装と仕様が矛盾する場合は実装を変更する。
- 仕様が不明確な場合は推測で埋めない。

### 2. 小さく変更する
- 変更はできるだけ小さく保つ。
- 1 回の変更は、原則として次のいずれか 1 つに絞る。
  - 1機能追加
  - 1バグ修正
  - 1リファクタリング
- 複数機能の同時実装や、機能追加と大規模リファクタリングの混在を避ける。

### 3. 可読性を最優先する
- コードは書く回数より読む回数の方が多い。
- 短さより、意図と責務が理解しやすいことを優先する。

### 4. AI は推測しない
- 実装前に仕様を読む。
- 実装前にテストを書く。
- 不明点は質問または未確定事項として残す。
- 大規模変更を一度に行わない。

## 実装手順
1. 仕様、設計、既存コード、既存テストを読み、変更対象と影響範囲を特定する。
2. 追加仕様またはバグ再現を表すテストを書く。
3. テストを失敗させ、仕様とテストの対応を確認する。
4. 最小実装でテストを通す。
5. 必要最小限のリファクタリングを行う。
6. 関連テスト、型チェック、Lint を実行する。
7. 仕様や利用方法に変更があれば、必要な文書を更新する。

## TDD

### Red → Green → Refactor
必ず次の順で進める。
1. テストを書く
2. テストを失敗させる
3. 最小実装で通す
4. リファクタリングする

### テストは仕様を表現する
- テストは実装詳細ではなく振る舞いを記述する。
- 良い例:
  - `it("期限切れのユーザーはログインできない")`
- 悪い例:
  - `it("returns false")`

### 最重要ロジックからテストする
優先順位:
```text
Domain Logic
↓
Application Logic
↓
Infrastructure
↓
UI
```

### バグ修正時は再現テストを先に書く
- バグ修正では、修正前に失敗するテストを追加する。

## 設計原則

### 単一責任
- 1つの関数、クラス、モジュールは 1 つの理由でのみ変更されるべきである。

### 依存方向を制御する
- 重要なビジネスルールは外部技術に依存してはならない。
- 少なくとも次を直接知らないようにする。
  - UI
  - DB
  - API
  - Framework

### 副作用を分離する
- Pure Logic と Side Effect を分離する。
- 良い例:
  - `calculatePrice()`
  - `saveOrder()`
- 悪い例:
  - `calculateAndSavePrice()`

### 境界を明確にする
- 外部との境界を明示する。
- 例:
  - `Controller`
  - `UseCase`
  - `Repository`
  - `External API`

### ファイルを肥大化させない
- 1 ファイルは原則として 500 行以内に保つ。
- 行数が大きくなりすぎる場合は、責務分離が不十分な可能性を疑う。
- その場合は、関数、クラス、モジュールの分割を検討する。
- 例外が必要でも、単に 1 ファイルへ詰め込むのではなく、分割できない理由を説明できる状態にする。

## TypeScript

### strict mode を使用する
- `strict` は必須。

### any を使わない
- `any` は使わない。
- 必要なら `unknown` を使い、境界で絞り込む。

### 型で不正状態を表現できなくする
- 良い例:
  ```ts
  type Status = "draft" | "published";
  ```
- 悪い例:
  ```ts
  type Status = string;
  ```

### Nullable を明示する
- `null` / `undefined` を曖昧に扱わない。

## 関数設計

### 関数は小さく保つ
- 目安は 20〜30 行以内。
- 長くなる場合は責務分割を検討する。

### 関数コメントは Google スタイルで書く
- 関数コメントを追加する場合は、原則として Google スタイルの docstring を使う。
- コメントは日本語で記載する。
- `Args`、`Returns`、`Raises` など、必要な節を明示する。
- すべての関数に機械的に長いコメントを付けるのではなく、読み手の理解に必要な関数へ過不足なく付ける。

### 関数コメントは処理手順ではなく目的と意図を書く
- コメントでは「この関数が何をしたいのか」「なぜこの関数が存在するのか」を優先して説明する。
- 実装の逐次処理をそのまま言い換える説明は避ける。
- 読み手が呼び出し側から見て責務と期待結果を把握できる内容にする。
- 実装詳細が複雑な場合でも、まずは設計上の責務、前提条件、副作用、返す意味を明確にする。

### 関数名は意図を表す
- 良い例:
  - `calculateInvoiceTotal`
- 悪い例:
  - `processData`

### Boolean フラグで分岐させない
- 悪い例:
  - `getUser(true)`
- 良い例:
  - `getActiveUser()`

### ネストを浅くする
- 早期 return を優先する。

## データ設計

### Primitive Obsession を避ける
- 重要な値は専用型や value object で表現する。

### 不変性を優先する
- 可能な限り immutable にする。

### 共有状態を最小化する
- グローバル状態や暗黙共有を避ける。

## React

### UI とロジックを分離する
- コンポーネントに複雑な業務ロジックを書かない。

### 状態は最小化する
- 計算で導出できる値は state にしない。

### Client Component を増やしすぎない
- 可能なら Server Component を優先する。

## テスト設計

### テストは独立させる
- 実行順序に依存させない。

### テストデータを共有しすぎない
- 必要に応じて Fixture Builder を使う。

### 実装詳細をテストしない
- 公開インターフェースと観測可能な振る舞いをテストする。

## リファクタリング

### テストがない状態でリファクタリングしない
- 必ず安全網を作ってから行う。

### 重複を削除する
- ただし、早すぎる抽象化は禁止する。

### 命名改善を優先する
- 多くの場合、最初に改善すべきは設計全体より名前である。

## エラーハンドリング

### 例外を握り潰さない
- 例外をそのまま上位へ `throw` して伝搬させるのではなく、原則として `Result` 型や明示的な失敗値へ変換して返す。
- 使い手側に広範な `try-catch` を要求する設計を避ける。
- 失敗理由は握り潰さず、ログへ残すか、適切なドメインエラー情報として `Result` に載せる。

### Result 型で失敗を返す
- 予期できる失敗は `throw` ではなく `Result` 型で表現する。
- 成功と失敗が型で区別できるようにする。
- 失敗時は、呼び出し側が分岐可能なエラーコードや理由を返す。
- 例外は、プロセス継続不能、ライブラリ境界、想定外障害など、通常フローで扱えない場合に限定する。

### ユーザー向けと開発者向けを分離する
- 内部情報をそのまま見せない。

## セキュリティ

### シークレットをコミットしない
- 必須。

### 入力値を信用しない
- 外部入力は必ず検証する。

### 最小権限原則
- 必要最小限の権限だけを付与する。

## やってはいけないこと
- 仕様未確認のまま実装を始めること
- テストを書かずにロジック変更すること
- `any` で型問題を回避すること
- 実装詳細に依存した brittle なテストを書くこと
- 機能追加と無関係な大規模整理を同じ変更に混ぜること
- 例外を握り潰して原因を隠すこと
- シークレットや内部情報をコード、ログ、テストデータへ埋め込むこと

## 完了条件
タスク完了時は、少なくとも次を満たす。
- テスト追加済み
- テスト成功
- 型エラーなし
- Lint エラーなし
- 仕様準拠
- 必要なドキュメント更新済み
- レビュー可能なサイズである

## 出力
- 何を変更したかを簡潔に説明する。
- 追加または更新したテストを明示する。
- 実行した検証項目を明示する。
- 未解決の制約や人間確認が必要な事項があれば残す。
