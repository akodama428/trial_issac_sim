# ADR Design

## 目的
アーキテクチャの設計、比較、または ADR 形式での設計判断記録が求められたときに使う。

## 先に読むもの
- `AGENTS.md`
- `docs/MASTER.md`
- `docs/PROJECT.md`

アーキテクチャ設計は、必ず `docs/PROJECT.md` に記載された要件に基づいて行う。

## 手順
1. まずリポジトリの指示と関連するプロジェクト文書 `PROJECT.md` を読む。
2. 要件がアーキテクチャ議論に混ざっている場合、設計を確定する前に `PROJECT.md` の更新、または更新提案を行う。
3. 要件が不足しておりアーキテクチャ設計ができない場合は、 `PROJECT.md` の更新、または更新提案を行う。決して、不足した要件を推測してアーキテクチャ設計を行わないこと。
4. 設計判断のドライバーを特定する。
5. 少なくとも次を整理する。
- ユーザーに見える機能
- 非機能制約
- 運用上の制約
- データソースと外部連携
- 実装コストと保守コスト
1. 少なくとも2案以上のアーキテクチャ案を提示する。
2. 各案について次を記載する。
- 主要コンポーネント
- 各コンポーネントの責務
- コンポーネント間の依存方向
- 主なリクエストフローまたはデータフロー
- 想定されるメリット
- 想定されるデメリット
- 主要リスク
1. 各案について、最低限次のクリーンアーキテクチャ原則を確認する。
- 単一責務の原則
- 依存関係ルール
1. それぞれの案が上記原則に対してどこが強く、どこが弱いかを明記する。
2.  推奨アーキテクチャを1つ選ぶ。
3.  設計判断ドライバーとトレードオフに基づいて、推奨理由を説明する。
4.  結果を `ADR.md` に記録する。

## 必須出力
案の列挙だけで終わらせず、必ず設計判断記録を作る。

ADR には少なくとも次を含める。
- タイトル
- ステータス
- 背景
- 設計判断ドライバー
- 検討した選択肢
- 選択肢比較
- 要件定義番号とモジュールの対応表
- 推奨案
- 推奨理由
- 影響
- フォローアップ項目

## 案比較のルール
- 常に2案以上を提示する。
- 抽象論ではなく、実コンポーネント、境界、データフローが分かる具体案にする。
- すべての案について、メリットとデメリットの両方を記載する。
- 制約上どうしても必要な場合を除き、最も複雑な案を安易に推奨しない。
- 現在の要件を満たしつつ、拡張余地がある最も単純な構成を優先する。

## やってはいけないこと
- 不足している要件を勝手に推測して、アーキテクチャを決めること。
- `docs/PROJECT.md` に書かれていない前提を既成事実として扱うこと。

不足している要件や検討観点がある場合は、それを明示したうえで、ユーザーに提案すること。

## クリーンアーキテクチャ確認項目

### 単一責務の原則
各主要コンポーネントが、ひとつの主たる変更理由だけを持つか確認する。

例:
- UI 描画コンポーネントが、リポジトリ走査ロジックまで持たないこと
- 検索インデックス生成処理が、HTTP レスポンス処理まで兼務しないこと
- ドメイン比較ロジックが、ページ描画都合に依存しないこと

### 依存関係ルール
依存関係が、フレームワークや変化しやすい詳細へ outward に向かうのではなく、より安定したポリシー層へ inward に向かっているか確認する。

最低限の確認項目:
- ドメインルールが UI コードに依存していない
- ドメインルールがインフラやフレームワーク詳細に直接依存していない
- インフラがドメインの契約に依存しており、その逆になっていない
- ビジネスルールが、アプリ操作、UI、DB、外部 API から分離されている

これらに違反する案がある場合は、ADR 内で明示的に指摘する。

## ADR 構成
リポジトリに既存の ADR 形式がない限り、次の構成を使う。

```md
# ADR-xxxx: <設計判断タイトル>

## Context
<課題、対象範囲、制約>

## Decision Drivers
- <ドライバー1>
- <ドライバー2>

## Options Considered
### Option 1: <案名>
- Overview:
- Responsibilities:
- Dependency direction:
- Merits:
- Demerits:
- Clean Architecture notes:

### Option 2: <案名>
- Overview:
- Responsibilities:
- Dependency direction:
- Merits:
- Demerits:
- Clean Architecture notes:

## Decision
<推奨案>

## Requirement to Module Mapping
| 要件番号 | 要件概要 | 主担当モジュール | 備考 |
| --- | --- | --- | --- |
| REQ-xxx | <要件の要約> | <モジュール名> | <必要なら記載> |

## Rationale
<なぜ現時点でこの案を推奨するか>

## Consequences
- Positive:
- Negative:

## Follow-up
- <次のアクション>
```

## リポジトリ固有の指針
- リポジトリに `docs/MASTER.md` がある場合は、設計判断前に必ず読む。
- 要件文書とアーキテクチャ文書が分かれている場合は、その分離を維持する。
- 既に `ADR.md` がある場合は、競合する別形式を増やさず、追記または更新する。
